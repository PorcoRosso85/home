# phase
/phase

# 説明
TDD原則に基づき、タスクをBaby Steps（15-30分単位）に分解。各フェーズは必ず「仕様→テスト→実装」の順序で構成。

# 実行内容

## 1. 仕様の明確化（/tested原則）
- 達成すべき要件を明文化
- その要件がアプリケーションにどう貢献するか確認
- テスト＝仕様として表現可能か検証

## 2. TDDサイクルでの分解（/tdd原則）
各Baby Stepは以下の構造を厳守：

```
Step X: [機能名]の実装
├─ X.1: 仕様をテストで表現（RED）
│  └─ 黄金律「リファクタリングの壁」チェック
├─ X.2: テスト失敗を確認
│  └─ 失敗メッセージが仕様を明確に示すか
├─ X.3: 最小実装（GREEN）
│  └─ YAGNI原則：必要最小限のみ
├─ X.4: 全体テスト実行
│  └─ nix run .#test
└─ X.5: リファクタリング（REFACTOR）
   └─ テストがGREENを維持
```

## 3. フェーズ生成ルール

### 新規開発（Beck流）
1. 仕様理解とテスト設計フェーズ
2. RED-GREEN-REFACTORサイクル
3. 統合確認フェーズ

### 既存コード改修（Feathers流）
1. 特性テスト作成フェーズ（現状保護）
2. 新仕様のテスト追加フェーズ
3. 実装改修フェーズ
4. リファクタリングフェーズ

## 4. 優先度設定
- **高**: テスト作成（仕様の明文化）
- **中**: 実装（テストを通す）
- **低**: リファクタリング（品質改善）

# Redフェーズチェックリスト（自動確認）
各テスト作成タスクで以下を確認：
- □ 要件・仕様を正確に反映
- □ 振る舞いを検証（実装詳細ではない）
- □ 失敗メッセージが明確
- □ Baby Stepsとして適切な粒度
- □ テストファイル配置が規約準拠

# 連携
- `/tdd`: 詳細なTDDプロセス参照
- `/tested`: テストの価値確認
- TodoWrite: 生成したフェーズを記録
- `/orch`: 独立したフェーズの並列実行
