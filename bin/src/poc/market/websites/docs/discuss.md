# 複数URI管理システム - 議論記録

## 議題1: サブドメイン vs サブディレクトリの使い分け

### 初期質問
「ドメイン分けなくてもサブドメイン（サブディレクトリではない）で分離すればそれでオールOK?」

### 提案
サブドメインでの分離は多くのケースで有効だが、3段階の拡張戦略を推奨：
- Phase 1: サブドメインのみ（api.example.com, admin.example.com等）
- Phase 2: サブドメイン＋パス（api.example.com/v1等）
- Phase 3: ハイブリッド（必要時のみ）

### 主要な反論と回答
1. **反論**: SEOの観点でサブドメインは別サイト扱いになり不利では？
   **回答**: Google は2022年以降サブドメインの評価を改善。適切な構造化データで対応可能。

2. **反論**: Cookieやセッション共有で問題が起きるのでは？
   **回答**: `Domain=.example.com` 設定で共有可能。セキュリティはOAuth2.0/SAMLで担保。

## 議題2: 責務分担と大規模URI設計

### 問題提起
「サブドメインとサブディレクトリの責務分担はある？」
「47都道府県のみならず、郵便番号単位での区分も検討している」

### 提案：責務境界に基づくURI設計原則
```
サブドメイン = 独立した実行コンテキスト
- api.example.com（独立したセキュリティ境界）
- static.example.com（CDN直結、Cookie無効）

サブディレクトリ = 共有コンテキスト内の名前空間
- /area/（データの論理区分）
- /service/（機能の論理区分）
```

### 郵便番号レベルの設計（12万件対応）
```
example.com/13/100/0001/  # 東京都/千代田区/皇居
```

### 反論と回答
1. **反論**: URLが`/13/163/8001/`では意味が分からない
   **回答**: パンくずリストで「東京都 > 新宿区 > 西新宿2丁目」と表示。開発者向けは数値、ユーザー向けは日本語で両立。

## 議題3: ローカライズ情報でのSEO差別化戦略

### 問題提起
「これをそもそも行う意義はあるか。ローカライズした情報で他とのSEO差別化を図ろうと思っている」

### 分析結果
LLM時代でも**リアルタイム性×超局所性×専門性**の組み合わせは有効：

1. **LLMの限界**
   - 「新宿区西新宿2-8-1の最寄りコンビニ」は答えられない
   - リアルタイムの工事情報、地域イベントは把握不可

2. **ビジネス価値**
   - 投資550万円に対し、4ヶ月で回収見込み
   - SEO以外にデータAPI販売も可能

## 議題4: データ主体SSGによる時事アーカイブ構想

### 提案内容
「データ主体のSSGを行うことで、ただただ時事ネタを積み上げていく新聞のようなもの」

### 具体的な実装
```
/13/100/0001/2024/01/23/
├── index.html    # その日の記録
├── weather.json  # 天気データ
├── events.json   # イベント
└── sponsor.json  # スポンサー情報
```

### 収益モデル
- 地域スポンサー: 月額1万円/郵便番号
- 開発者広告: 技術ショーケース
- データAPI: 月額10万円

### 主要な反論と回答
1. **反論**: 時事ネタは鮮度が命。過去のニュースに価値はあるのか？
   **回答**: 「2020年3月15日の渋谷」など特定日付×場所の記録は歴史的価値を持つ。不動産購入時の地域調査等で需要。

2. **反論**: 毎日12万ページ生成はコスト的に非現実的では？
   **回答**: 段階的展開。主要100地点から開始し、アクセスログから需要を特定して拡大。

3. **反論**: 地域企業がスポンサーになる動機が弱い？
   **回答**: 「地域貢献の可視化」として訴求。CSRレポートにも記載可能な社会貢献活動。

4. **反論**: 著作権問題でコンテンツ作成が困難？
   **回答**: 事実データ（気温、統計）中心。ニュースは見出しとリンクのみ。公開データを活用。
