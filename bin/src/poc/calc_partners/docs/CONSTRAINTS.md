# 制約事項

## 絶対禁止リスト

### 1. UI/UXの過剰な装飾
**禁止事項**:
- CSSアニメーション
- カスタムUIコンポーネント
- トランジション効果
- パララックススクロール
- ローディングスピナーの凝った演出

**許可**:
- Bootstrap標準コンポーネントのみ
- 必要最小限のツールチップ

### 2. CRM機能の実装
**禁止事項**:
- パートナー連絡先管理
- 商談進捗管理
- タスク管理
- カレンダー機能
- リマインダー

**代替**:
- 既存CRMツールとのAPI連携のみ

### 3. コミュニケーション機能
**禁止事項**:
- チャット機能
- コメント機能
- 通知システム
- メール送信
- ダッシュボード内メッセージング

**代替**:
- Slack/Teams等の外部ツール利用を前提

### 4. 過剰なオンボーディング
**禁止事項**:
- 動画チュートリアル
- インタラクティブツアー
- ステップバイステップガイド
- ポップアップヘルプ
- ウィザード形式の設定

**許可**:
- 静的FAQページ1枚のみ

### 5. 個別カスタマイズ
**禁止事項**:
- 顧客別の特別機能
- カスタムレポート形式
- 個別の計算ロジック変更
- UIのカスタマイズ要求対応
- 特定顧客向けAPI

**原則**:
- プロダクトの型に顧客を合わせる

## 開発時の判断基準

### 機能追加の判断フロー
```
新機能要求
    ↓
計算・シミュレーションに直接関係するか？
    NO → 却下
    YES ↓
既存の外部ツールで代替可能か？
    YES → 却下
    NO ↓
実装工数 < 価値創出か？
    NO → 却下
    YES → 検討可能
```

### コード品質基準

#### 必須
- 計算ロジックの単体テスト（カバレッジ100%）
- 型定義の完全性
- エラーハンドリング

#### 不要
- UI単体テスト
- E2Eテスト（最小限）
- パフォーマンステスト（計算部分以外）

## 技術選定の制約

### 使用禁止
- 複雑なステート管理ライブラリ（Redux等）
- UIフレームワークの重複使用
- 非標準的なビルドツール

### 必須要件
- **nix develop必須**: すべてのコマンドは`nix develop`環境内で実行
- **nix shell必須**: 一時的なツール使用時は`nix shell nixpkgs#<package>`形式を使用
- **グローバルコマンド禁止**: `node`, `npm`, `pnpm`等の直接実行は禁止
- **環境外実行の検証**: nix環境外でのコマンド実行を発見した場合は即座に修正

### 推奨
- 標準ライブラリの活用
- 軽量な依存関係
- メンテナンスが容易な技術

## レビュー観点

### 必ずチェック
1. 計算ロジックの正確性
2. API連携の堅牢性
3. データの一貫性

### チェック不要
1. UIの美しさ
2. UXの滑らかさ
3. カスタマイズ性

## 例外処理

### 許可される例外
なし。例外を作らない。

### エスカレーション
顧客要求が規約に反する場合:
1. 明確に「NO」と伝える
2. 代替案（外部ツール）を提示
3. それでも要求される場合は契約終了

## メンテナンス方針

### やること
- 計算ロジックのバグ修正
- API仕様変更への対応
- セキュリティパッチ適用

### やらないこと
- UI改善要求への対応
- 機能追加要求への対応
- カスタマイズ要求への対応

この規約は絶対である。議論の余地はない。
