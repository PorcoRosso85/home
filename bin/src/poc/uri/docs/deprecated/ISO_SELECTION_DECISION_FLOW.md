# 判断フロー：あなたのアプリケーションに最適なISOを見つける方法

以下の質問に上から順番に答えていくことで、参照すべきISO標準の組み合わせを特定できます。

## ステップ1：ドメイン特化型（最も具体的で強力）

**質問： あなたのアプリケーションは、特定の規制産業や、業務プロセスが高度に標準化されている業界向けですか？**

### [YES] の場合
まずは、そのドメインに特化したISO規格を主たる参照元とします。これがLevel 1のフィーチャーを定義するための最も直接的な「カタログの元ネタ」になります。

| 業界ドメイン | 最優先で参照すべきISO規格（例） |
|------------|--------------------------------|
| 自動車 | ISO 26262 (機能安全), ISO/SAE 21434 (サイバーセキュリティ) |
| 医療機器 | ISO 13485 (品質マネジメント), ISO 14971 (リスクマネジメント), IEC 62304 (ソフトウェアライフサイクル) |
| 金融サービス | ISO 20022 (金融通信メッセージ) |
| 航空宇宙 | ARP4754A/ED-79A, DO-178C/ED-12C (注: これらはISOではないが、業界のデファクトスタンダード) |
| 鉄道 | EN 50128 (注: 欧州規格だが、国際的に広く参照される) |
| 情報セキュリティ | ISO/IEC 27001 (ISMS) の Annex A は、セキュリティ機能の優れたチェックリストとして機能します。 |

### [NO] の場合、または上記に加えて
ステップ2に進みます。

## ステップ2：汎用マネジメントシステム型（アプリケーションの「あり方」を定義）

**質問： ドメイン特化の規格がない場合、あなたのアプリケーションにとって最も重要な管理側面はなんですか？**

これは、特定の機能リストというよりは、アプリケーション全体が準拠すべき**「非機能要件」や「運用機能」**の元ネタとなります。

| 管理側面 | 参照を検討すべきISO規格 | アプリケーション例 |
|---------|---------------------|----------------|
| 情報セキュリティ | ISO/IEC 27001 (ISMS) | ほぼ全てのアプリケーション。特に個人情報や機密情報を扱う場合。 |
| サービスとしての提供 (SaaS等) | ISO/IEC 20000-1 (ITサービスマネジメント) | クラウドサービス、APIプラットフォームなど。 |
| 品質保証プロセス | ISO 9001 (品質マネジメント) | 顧客への品質コミットメントが重要なBtoB製品など。 |
| 個人情報保護 | ISO/IEC 27701 (PIMS) | GDPRやAPPIなど、地域のプライバシー法規制への対応が必要な場合。 |

## ステップ3：基礎・土台となる規格（全てのアプリケーションに推奨）

**質問： 上記に関わらず、全てのソフトウェア開発の土台として参照すべき、共通言語となる規格はありますか？**

はい、あります。これらは具体的なフィーチャーリストではありませんが、フィーチャーを定義し、評価するための**「共通の物差し」**を提供してくれます。

| 目的 | 参照すべき基礎規格 | グラフへの影響 |
|-----|----------------|------------|
| 品質特性の定義 | ISO/IEC 25010 (SQuaRE) | 全てのLevel 1ノードに付随する**非機能要件（性能、信頼性、使用性など）**を定義するための語彙となります。Level 2や3で参照されます。 |
| 開発プロセスの定義 | ISO/IEC/IEEE 12207 | 「構成管理機能」「デプロイメント機能」といった、開発・運用を支える**支援機能（フィーチャー）**を洗い出すためのフレームワークとなります。 |

## 結論：最適なアプローチは「ハイブリッド（組み合わせ）」

現実のプロジェクトでは、単一のISOを参照して終わり、ということは稀です。

### 例：「新しい医療系SaaSアプリケーション」を開発する場合

参照すべきISOは以下のようになります。

#### 1. ISO 13485 / IEC 62304（ドメイン特化）
医療記録の管理、診断支援、処方箋発行といった、中核となるLevel 1フィーチャーを定義するための主たる参照元。

#### 2. ISO/IEC 27001 / 27701（マネジメント）
患者データという機微情報を守るための、アクセス制御、監査ログ、暗号化といったセキュリティ・プライバシー関連のフィーチャーと非機能要件を定義。

#### 3. ISO/IEC 25010（基礎）
全てのフィーチャーに対して、「性能効率（レスポンスタイム）」「信頼性（ダウンタイム）」「使用性（分かりやすさ）」といった品質目標を設定するための共通言語として使用。

このように、**ドメイン特化の規格で骨格を作り、汎用規格で肉付けし、基礎規格で品質を定義する**というハイブリッドアプローチが、最も安全かつ網羅的な方法と言えるでしょう。
