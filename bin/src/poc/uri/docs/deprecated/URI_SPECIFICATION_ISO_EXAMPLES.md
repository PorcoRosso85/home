# URI Specification - ISO標準を活用したLevel 1機能定義

## 1. ドメイン固有のISO規格（Level 1のフィーチャーカタログ）

特定の業界では、業務プロセスやデータ交換の形式がISOによって標準化されています。その規格の構成要素が、そのままLevel 1の機能（フィーチャー）の定義に直結します。

### 金融業界: ISO 20022 (金融通信メッセージ)

**概要**: 銀行間の支払、証券取引、カード取引などのためのメッセージ形式を定義した、世界標準です。

**カタログとしての使い方**:
- 規格で定義されている「pacs.008 (顧客送金指図)」というメッセージは、アプリケーションの**「送金指図機能 (.../features/payment-initiation)」**に直接対応します。
- 「camt.053 (口座取引明細通知)」は**「取引明細照会機能 (.../features/account-statement)」**に対応します。
- このように、規格の構造を解析することで、そのドメインで必要とされる機能リストを体系的に洗い出すことができます。

### 医療情報学: ISO 13606 (電子健康記録の通信)

**概要**: 電子カルテの情報を異なるシステム間で交換するための標準モデル。

**カタログとしての使い方**:
- この規格のデータ構造（処方箋、検査結果、アレルギー情報など）から、「処方箋管理機能」「検査結果参照機能」といったフィーチャーを定義できます。

## 2. ソフトウェア工学全般のISO規格（機能定義のフレームワーク）

これらは機能リストそのものではありませんが、「どのような観点で機能を洗い出すべきか」というフレームワークを提供してくれます。

### ISO/IEC/IEEE 15288 (システムライフサイクルプロセス)

**概要**: システムの構想から廃棄まで、どのようなプロセスが必要かを定義しています。

**カタログとしての使い方**:
- この規格の「技術プロセス」の章には、「アーキテクチャ定義」「設計定義」「検証」「妥当性確認」などが含まれます。これらから、「（開発者向けの）監査ログ機能」や「（運用者向けの）システム監視機能」といった、ビジネス機能ではないが重要な「システム機能」を導出するヒントになります。

### ISO/IEC 25010 (SQuaRE - 品質モデル)

**概要**: ソフトウェアの品質を定義するためのモデルです。

**カタログとしての使い方**:
- 品質特性の一つである**「機能性 (Functional Suitability)」**の章は、「機能がどの程度、明示的・暗示的なニーズを満たすか」を定義しています。
- この中のサブ特性「機能網羅性」や「機能正確性」を評価するプロセスで、「本来あるべきだった機能（フィーチャー）」の抜け漏れを発見することができます。これは、カタログそのものというより、カタログをレビューするためのチェックリストとして機能します。

## ASCIIアートの更新（Level 1のカタログ候補にISOを追加）

これらの議論を反映し、Level 1のカタログ候補に「ドメイン固有のISO規格」を追加するのが最も正確です。

```
+--------------------------------+
| Level 0: Business Capability   |
+--------------------------------+
       +-------------------------------+
       | Capability: Customer Management |
       |(.../capabilities/cust-mgmt)   |
       +-------------------------------+
                     |
         "支援する (SUPPORTS)"
                     |
+--------------------------------+
| Level 1: Feature (Domain/Service)|  <--- WHAT: ドメインの機能分割。
| (カタログ候補: DDD, Service Patterns, |
|              Domain-specific ISOs)   |  <- ISOを追加
+--------------------------------+
       +------------------------------------+
       | Feature: Payment Initiation        |  <- ISO 20022由来の例
       | (.../features/payment-initiation)  |
       +------------------------------------+
                   |
     "要求カテゴリを持つ (HAS_REQUIREMENTS_IN)"
                   |
+--------------------------------+
| Level 2: Classification        |
+--------------------------------+
       +---------------------------+
       |   Category: ASVS V4 Txn Auth|  <- ASVSの「トランザクション認証」章
       |   (.../asvs/v4.0.3/v6)    |
       +---------------------------+
                   |
           "含む (CONTAINS)"
                   |
+--------------------------------+
| Level 3: Concrete Requirement  |
+--------------------------------+
   +-------------------------------+
   | V6.2.2: Re-authentication for |
   | sensitive transactions.       |
   +-------------------------------+
```

**変更点の説明:**

- **Level 1のカタログ候補に`Domain-specific ISOs`を追加:** これで、DDDのようなデファクトスタンダードと、ISOのようなデジュールスタンダードの両方を視野に入れていることが明確になりました。
- **Level 1のノード例を変更:** よりISOとの関連が分かりやすいように、金融ドメインの「支払指図機能」を例にしました。
- **下位レベルの例も更新:** 「支払指図」という金融取引に関連が深い、ASVSのV6章「トランザクション認証/整合性」を例として使用し、全体の整合性を高めました。

## 結論

汎用的な機能カタログとしてのISOは存在しませんが、**あなたのアプリケーションが属するドメインに特化したISO規格があれば、それはLevel 1の機能を定義するための、最も信頼性が高く、体系的な「カタログの元ネタ」となり得ます。**
